# v1.0 Release Checklist

This checklist defines what must be true before calling Autonomous Enterprise AI OS portfolio-ready
v1.0. It is intentionally broader than test pass/fail because the release needs to be understandable,
demoable, and deployable by a reviewer.

## Release Candidate

- [ ] `main` is green in GitHub Actions.
- [ ] The release branch has no unmerged product tickets.
- [ ] `pyproject.toml` version is intentionally set for the release candidate.
- [ ] A GitHub release draft links to these notes and the demo package.
- [ ] The final commit SHA is recorded in the release notes.

## Validation

- [ ] `make lint`
- [ ] `make test`
- [ ] `make smoke`
- [ ] `make demo`
- [ ] `make k8s-validate`
- [ ] `make cloud-validate`
- [ ] `make production-validate`
- [ ] `make release-package`

## Documentation

- [ ] README links to portfolio architecture, development guide, Kubernetes, AWS, release checklist,
  demo walkthrough, and release notes.
- [ ] `docs/portfolio-architecture.md` explains the platform without requiring codebase reading.
- [ ] `docs/development.md` covers local setup, API use, demo execution, persistence, queueing,
  security, observability, and deployment paths.
- [ ] `deploy/kubernetes/README.md` covers base/local/staging/production overlays and required
  secrets/config.
- [ ] `deploy/cloud/aws/README.md` covers AWS permissions, Terraform, secret sync, migrations,
  rollout, smoke checks, and teardown.

## Product Surfaces

- [ ] `/app` can create and browse runs.
- [ ] `/app/artifacts` can inspect generated artifacts, lineage, previews, and download links.
- [ ] `/app/admin` can inspect agents, connectors, credential profiles, policy rules, and affected
  runs.
- [ ] `/run-inspector/runs/{run_id}` can inspect graph nodes, events, timelines, approvals,
  deployments, evaluations, and lineage.
- [ ] `/docs`, `/health`, and `/metrics` are reachable.

## Demo Flow

- [ ] `examples/procurement_demo.csv` is the canonical demo dataset.
- [ ] `make demo` produces a completed run.
- [ ] Demo output includes `demo_summary.json`, `metrics.prom`, dashboard, report, chart, KPI, code,
  and evaluation artifacts.
- [ ] Demo walkthrough explains how to inspect the generated run in the UI.
- [ ] Demo package generated by `make release-package` contains README, checklist, release notes,
  portfolio architecture, demo walkthrough, sample dataset, and validation manifest.

## Security And Governance

- [ ] Tool permission registry includes read-only, write, external-network, code-execution, and
  deployment categories.
- [ ] Destructive tools are blocked by default.
- [ ] External connector and deployment actions require approval/escalation.
- [ ] Auth-enabled mode gates protected API actions by role.
- [ ] Secret-like payloads are redacted from API responses and run archives.

## Observability

- [ ] Every run has a trace ID.
- [ ] API requests, workflow execution, agent nodes, policy decisions, and evaluations create trace
  spans when tracing is enabled.
- [ ] `/metrics` exposes run, artifact, evaluation, retry, duration, workflow-job, and node status
  metrics.
- [ ] Grafana dashboard can be loaded from `deploy/grafana/provisioning/dashboards`.
- [ ] Optional MLflow and LangSmith adapters are documented.

## Deployment

- [ ] Docker Compose starts API, worker, Postgres, Redis, MinIO, Prometheus, and Grafana.
- [ ] Kubernetes base and local/staging/production overlays pass `make k8s-validate`.
- [ ] Runtime config validator fails clearly when required production secrets/config are missing.
- [ ] AWS Terraform package passes `make cloud-validate`.
- [ ] AWS deployment path identifies EKS, RDS, ElastiCache, S3, Secrets Manager, IAM, and VPC
  resources.
- [ ] TLS/WAF ingress, autoscaling, disruption budgets, network policies, SLO alerts, and runbooks
  pass the production readiness validator.
- [ ] Staging load, security, failure-injection, backup restore, and 24-hour soak evidence is attached.
- [ ] Versioned launch and rollback commands have been exercised against the release candidate.

## Final v1.0 Acceptance Criteria

- [ ] A reviewer can understand the architecture from README and `docs/portfolio-architecture.md`.
- [ ] A reviewer can run the procurement demo from clean setup instructions.
- [ ] A reviewer can inspect generated artifacts and run state through the UI.
- [ ] A reviewer can see production-shaped deployment paths for Docker Compose, Kubernetes, and AWS.
- [ ] Release notes clearly explain what the platform demonstrates and what is intentionally out of
  scope.
